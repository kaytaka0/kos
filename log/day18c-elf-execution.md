# ELF実行時のクラッシュトラブルシューティングと解決策 (Day 18c 進行記録)

## 1. 問題の概要
OS (`terminal.cpp`) 上で ELF 形式のアプリケーション (例: `rpn`) をロードして実行しようとすると、`auto f = reinterpret_cast<Func*>(entry_addr); f();` が呼ばれた瞬間にシステムがクラッシュし、QEMU（仮想マシン）が意図せずリセット（トリプルフォルト）を繰り返す現象が発生した。
day18c で記載されているコード例ではどうやら ELF を正しく実行できなさそう。

## 2. 原因調査と解析 (`readelf` コマンドからの判明事項)
`readelf -l` (プログラムヘッダ) および `readelf -S` (セクションヘッダ) を用いて `rpn` アプリのバイナリを解析した結果、以下の2つの致命的な問題が連鎖してクラッシュを引き起こしていることが判明した。

### 原因①: `Offset` と `VirtAddr` のズレによる不正な命令の実行
*   **現象:** `e_entry` に記載されているエントリーポイントは `VirtAddr` (メモリ展開後の仮想アドレス基準) であるが、ファイル内のバイナリ配置である `Offset` とは一致していないことが多い。
*   **解析:** `rpn` バイナリの `.text` セグメントは、`Offset` が `0x160` から始まっているのに対し、`VirtAddr` は `0x1160` であった（0x1000のズレ）。
*   **影響:** 本来はメモリレイアウトを再構成してから実行しなければならないが、OS側は「ファイルデータをそのままメモリ (`file_buf`) に置き、先頭に `e_entry` の値を足すだけ」としていた。その結果、まったく関係のない「セクションヘッダなどの文字データ群」をCPUに実行させようとしてしまい、1歩目で不正な命令例外 (`#UD`) を引き起こして即死していた。

### 原因②: `.bss` 領域（未初期化変数）への書き込みによるバッファオーバーラン（カーネル破壊）
*   **現象:** C++ ソース内の初期化されていないグローバル変数 / 静的変数 (`long stack[100];` や `int stack_ptr;` など) は `.bss` セクションとして扱われ、ファイル上には容量を持たない（`NOBITS`）が、実行時にはメモリ空間を要求する。
*   **解析:** プログラムヘッダのデータセグメント (RWフラグ) において、ファイル上のサイズ (`FileSiz`) は 0 または小さい値であるにもかかわらず、メモリ確保要求サイズ (`MemSiz`) はそれより 816 バイト大きく指定されていることが確認できた。これが `.bss` の要求分である。
*   **影響:** 既存の OS 実装は、ファイルの容量分 (`FileSiz` の総和相当) しか実行用メモリを OS 側で確保していなかった。そのため、アプリが `.bss` に該当する変数への書き込みを行った途端、OS がアプリ用に確保したバッファの末尾を突き抜け、**その直後に存在していたカーネルの重要な管理領域や他プロセスのメモリを上書き破壊**してしまう。

## 3. 解決策 (ELF セグメントローダーの実装)
ファイルの生データをそのまま実行するという手法を破棄し、プログラムヘッダ (`Elf64_Phdr`) の指示に従って正しい間隔・サイズでメモリへの展開と初期化を行う「ELF ローダー処理」を `terminal.cpp` の `ExecuteFile` 内部に実装する。

**実装手順:**
1.  **展開に必要な全体サイズの計算:** `PT_LOAD` を持つプログラムヘッダをループし、最も小さい仮想アドレス (`first`) と最も大きい仮想アドレスの末尾 (`last`) を探し、アプリ全体が必要とする本番メモリの幅 (`last - first`) を計算する。
2.  **安全なメモリ領域の確保:** 計算した幅に基づき、アプリの実行環境となるメモリバッファ (`app_memory` 等) を新しく確保する。これにより `.bss` 分まではみ出ても安全な物理空間が確保される。
3.  **セグメントの展開 (コピー):** 各 `PT_LOAD` セグメントの `p_offset` の位置から実データ (`p_filesz` バイト) を、確保したメモリ空間の `p_vaddr - first` (ズレ・間隔を維持した相対位置) へ `memcpy` する。
4.  **`.bss` の初期化 (一番重要):** 各セグメントの末尾の確保領域、すなわち `p_memsz - p_filesz` バイトの空間に対して、`memset` で `0` (ゼロクリア) を行う。
5.  **エントリーポイントの再設定:** `e_entry` の値を展開先メモリベースからの相対距離に補正し、正しくジャンプ (`f()`) させる。

これにより、アドレスズレを吸収しつつ、`.bss` によるカーネル破壊を完全に防いで ELF アプリを実行できるようになった。
